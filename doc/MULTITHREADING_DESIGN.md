# MonsterWar 多线程设计说明

> 目的：为后续简历展示与代码演进提供一份可落地的多线程方案说明。
>
> 原则：**只把“可以并行”的工作放到后台线程，所有会改动 ECS、渲染器、UI 和场景栈的操作仍然保留在主线程。**

> 现状：项目已开始落地该方案，`GameApp` 会异步解析资源映射，`GameScene` 启动阶段会并行初始化关卡配置、存档数据和 UI 配置，关卡文件解析已独立为后台任务，保存文件写入也已切换为异步任务。

---

## 1. 适合多线程的区域

### 1.1 资源预加载

当前 `ResourceManager` 已通过 `assets/data/resource_mapping.json` 统一管理纹理、音频和字体路径，适合在场景切换前做后台预加载。

可并行的内容：
- 纹理缓存预热
- 音频资源预读
- 字体资源预读
- 关卡依赖资源的批量扫描

适用位置：
- `ResourceManager::loadResources()`
- `GameApp` 初始化阶段
- `TitleScene` → `GameScene` 切换前的等待阶段

建议：
- 后台线程只负责“读取文件与构建临时结果”
- 主线程负责“提交到 `ResourceManager` 缓存”
- 不直接在后台线程调用 SDL 渲染、ImGui、场景切换或 ECS 写操作

### 1.2 关卡 JSON 解析

`LevelLoader::loadLevel()` 需要读取 `.tmj` / `.tsj` 文件并构建图层数据。这个阶段很适合拆成：
1. 后台线程读取 JSON
2. 主线程根据解析结果创建场景实体

当前代码中以下内容可考虑后台化：
- 关卡文件读取
- tileset 文件读取
- 图层元数据整理
- 对象图层数据预处理

适用位置：
- `LevelLoader::loadLevel()`
- `GameScene::loadLevel()`

当前实现已将关卡文件与 tileset 文件读取拆分为 `LevelLoader::loadLevelDataAsync()`，`GameScene::init()` 会在拿到关卡路径后提前启动解析任务，再在主线程把解析结果提交到场景中。

### 1.3 存档与结果文件写入

`assets/save/SLOT_1.json` 这类写回操作不影响实时战斗，适合异步执行。

可并行的内容：
- 存档序列化
- 结果面板数据落盘
- 调试统计输出文件

适用位置：
- `GameScene::onSaveRequested()`
- 未来的通关结算 / 失败结算流程

### 1.4 非实时数据计算

以下逻辑在未来如果规模扩大，也可以独立成后台任务：
- 路径点网格预计算
- 波次队列整理
- 大量单位的数值预演
- 调试统计与性能采样聚合

---

## 2. 不建议多线程的区域

以下逻辑当前应保持主线程执行：
- `entt::registry` 的增删改
- `SceneManager` 的场景栈操作
- `Renderer` 与 `ImGui` 渲染
- `InputManager` 的 SDL 事件处理
- UI 控件创建与销毁
- 音乐播放、音效播放入口

原因：
- 这些模块依赖 SDL 或 UI 上下文
- ECS 写操作容易产生竞争条件
- 场景切换必须保持顺序确定性

---

## 3. 推荐线程模型

### 3.1 最小侵入方案

推荐先采用 **一个后台任务队列 + 一个主线程提交队列**：

- 后台线程负责：
  - 文件读取
  - JSON 解析
  - 资源路径扫描
  - 大批量数据预处理
- 主线程负责：
  - 资源注册
  - 场景创建
  - ECS 写入
  - UI 更新

### 3.2 生命周期建议

可选实现方式：
- `std::jthread` 作为工作线程
- `std::stop_token` 用于退出控制
- `std::mutex + std::condition_variable` 管理任务队列
- `std::future` / `std::promise` 用于一次性加载任务

建议规则：
1. 场景初始化时提交加载任务
2. 主线程每帧轮询完成状态
3. 任务完成后一次性提交结果
4. 场景关闭时发出停止信号并等待回收

## 4. 简历可写的实现点

如果后续把这份方案落到代码里，简历上可以写成：

- 设计并实现资源预加载线程，降低场景切换卡顿
- 设计并实现资源映射异步解析与主线程预加载，降低启动阶段阻塞
- 将关卡 JSON 解析与资源扫描拆分为后台任务
- 引入主线程提交队列，避免 ECS 与渲染上下文并发写入
- 统一后台任务退出流程，保证场景切换与应用关闭安全

## 5. 测试保留与验证建议

### 5.1 保留现有测试

当前项目已有构建和调试验证流程，新增多线程内容时应保持：
- 现有编译通过
- 现有场景切换不回归
- 现有战斗、UI、音频播放不受影响

### 5.2 建议新增的测试项

#### 功能回归
- 重复进入 / 退出 `TitleScene`
- 重复进入 / 通关 / 返回 `LevelClearScene`
- 快速切换关卡时不崩溃
- 在资源尚未完成时切场景不崩溃

#### 并发稳定性
- 连续触发资源预加载任务
- 在后台加载中关闭窗口
- 在后台加载中重复点击开始游戏
- 在后台写存档时切换场景

#### 性能验证
- 记录场景切换前后的卡顿时间
- 统计资源预加载前后的首帧耗时
- 比较主线程阻塞时间

### 5.3 推荐放置位置

如果需要落成自动化或半自动测试，可优先放在：
- `PerformanceTests1`
- 现有 Visual Studio Test Explorer 用例
- 关卡切换与资源加载的集成验证脚本

---

## 6. 当前结论

结合现有代码结构，**最值得先做的多线程内容**是：

1. 资源映射异步解析与预加载
2. 关卡 JSON 解析
3. 存档写入

这三类任务和实时 ECS 更新解耦最强，也最适合做成简历中的“多线程优化”亮点。
