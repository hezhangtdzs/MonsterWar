# MonsterWar 渲染后端迁移计划

> 目标：在保持现有 `Scene`、`ECS`、`ResourceManager` 和 `UI` 结构不大改的前提下，把当前基于 SDL 内置渲染的后端逐步迁移到 `OpenGL`。  
> 结论先行：**优先迁移到 OpenGL，不建议直接跳到 Vulkan**。对当前项目来说，OpenGL 的改造成本更低，且更容易复用现有 2D 渲染流程。

---

## 1. 当前渲染边界

当前代码中，渲染职责主要分布如下：

- `GameApp`
  - 创建窗口与渲染上下文
  - 初始化 `Renderer`、`TextRenderer`、`ImGui`
  - 控制主循环中的 `render()` 调用

- `Renderer`
  - 负责精灵、UI 图片、矩形、圆形、相机空间转换和视口裁剪
  - 当前直接封装了 SDL 的绘制 API

- `TextRenderer`
  - 负责字体绘制
  - 当前依赖 SDL_ttf 和 SDL 渲染路径

- `UI` / `Scene`
  - 依赖 `Renderer` 提供的绘制接口
  - 不应直接依赖底层图形 API

因此，迁移时的重点不是重写整个游戏逻辑，而是**替换 `Renderer` 底层实现**，让上层调用保持稳定。

---

## 2. 推荐迁移顺序

### 阶段 1：渲染边界抽象

目标：让上层逻辑不直接依赖 SDL 渲染对象。

建议动作：

- 保留 `Renderer` 作为对外接口
- 将底层绘制实现抽成后端类
- 把窗口创建、图形上下文创建和帧提交逻辑与业务渲染分离
- 明确 `GameApp` 只管理生命周期，`Renderer` 只负责绘制

产出：

- 渲染后端接口雏形
- SDL 版后端可继续工作
- 后续可以无痛加入 OpenGL 后端

### 阶段 2：OpenGL 后端接入

目标：实现一个可运行的 OpenGL 后端，并保持现有 2D 画面显示正常。

建议动作：

- 由 SDL 创建窗口，但改为创建 OpenGL Context
- 完成基础清屏、present、纹理绘制、矩形绘制
- 将纹理上传逻辑从 SDL 绘制改为 OpenGL 纹理对象
- 保持相机、Sprite、UI 的调用接口不变

产出：

- `OpenGLRendererBackend`
- 可以渲染当前地图、单位、UI
- 现有场景逻辑不需要大改

### 阶段 3：文本与 UI 迁移

目标：让字体渲染和 ImGui 同时适配新后端。

建议动作：

- 检查 `TextRenderer` 是否仍依赖 SDL_Renderer
- 若需要，将文本生成转成纹理后交给 OpenGL 绘制
- 调整 ImGui 后端初始化为 OpenGL + SDL 组合
- 确保 HUD、菜单、调试面板功能不回退

### 阶段 4：清理 SDL 绘制依赖

目标：清理残留的 SDL 渲染调用。

建议动作：

- 检查 `Renderer`、`TextRenderer`、`GameApp` 中是否还有 `SDL_Renderer` 直连调用
- 统一资源上传、销毁和错误日志
- 确保未来替换 Vulkan 也只需要新增后端而不是重写上层逻辑

---

## 3. 为什么优先 OpenGL

### OpenGL 的优势

- 改造成本低于 Vulkan
- 与 SDL 窗口和输入系统配合成熟
- 适合当前项目的 2D 精灵、HUD、UI、特效渲染需求
- 更容易快速验证 demo 效果

### Vulkan 的问题

- 初始代码量大很多
- 需要更复杂的同步、命令缓冲、资源生命周期管理
- 对当前项目来说收益偏小
- 更适合在已经有渲染抽象层的基础上再推进

---

## 4. 风险点

### 4.1 文本与 UI 适配

`TextRenderer`、ImGui 和 OpenGL 后端的组合需要单独验证，避免 UI 一次性崩掉。

### 4.2 资源上传方式变化

SDL 纹理与 OpenGL 纹理对象的生命周期和销毁方式不同，需要统一资源管理策略。

### 4.3 现有渲染接口回归

目前项目已经大量调用 `Renderer::drawSprite()`、`drawUIImage()`、`drawFilledCircle()` 等接口，迁移时必须保持这些 API 稳定。

### 4.4 性能回归

换后端后要重点比较：

- 首帧耗时
- 单位数量增加后的帧时间
- 特效和 UI 同屏渲染时的性能

---

## 5. 建议的验证点

每个阶段都应验证：

- 编译通过
- 标题场景可以进入关卡
- 关卡地图可以正常显示
- 单位创建、战斗、技能、特效仍然正常
- UI、暂停、调试面板仍然可用
- 音频和保存功能不受影响

---

## 6. 推荐落地策略

建议按以下优先级执行：

1. **先做渲染后端抽象**
2. **再接 OpenGL 后端**
3. **最后清理 SDL 渲染残留**

如果目标只是把这个项目继续作为 demo 展示，`OpenGL` 已经足够；  
如果目标是学习图形管线，再在 OpenGL 稳定后考虑 `Vulkan`。
