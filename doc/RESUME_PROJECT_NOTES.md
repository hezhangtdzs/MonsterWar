# MonsterWar 简历素材与项目亮点

> 用途：给简历、作品集、面试讲解准备一份可直接复用的项目总结。  
> 定位：这是一个基于 `SDL3 + EnTT + C++23` 的 2D 塔防 demo，适合突出 **ECS 架构、数据驱动、事件驱动、性能量化、工程化落地**。

---

## 1. 项目一句话概述

MonsterWar 是一个自研的 2D 塔防练手项目，采用 `ECS` 架构和数据驱动资源配置，实现了从标题页、关卡进入、单位建造、波次战斗、技能释放、胜负结算到保存读档的单机闭环。

---

## 2. 简历里可以直接写的项目亮点

### 2.1 架构与工程设计

- 采用 `ECS` 将实体、组件、系统解耦，降低战斗、渲染、UI、音频之间的耦合度。
- 使用 `entt::dispatcher` 构建事件驱动链路，实现攻击、治疗、技能、死亡、波次等系统间通信。
- 使用 `BlueprintManager` + `EntityFactory` 实现数据驱动的单位创建，支持敌人、玩家、技能、投射物、特效的统一装配。
- 将关卡、单位、技能、UI 配置拆分到 `assets/data/*.json`，避免硬编码。
- 通过场景系统封装 `TitleScene`、`GameScene`、`LevelClearScene`，实现明确的场景生命周期。

### 2.2 游戏玩法闭环

- 完成塔防核心闭环：建造准备、放置确认、升级、出售、波次刷怪、基地扣血、胜负切换。
- 实现英雄技能系统：支持主动技能、被动技能、持续时间、冷却时间、技能回切动画。
- 实现阻挡与近战交互：敌人被阻挡时可正常攻击阻挡者，并在敌人死亡后解除阻挡状态。
- 实现远程投射物与治疗特效：远程单位在命中帧触发投射物，治疗单位触发恢复事件。
- 完成保存流程：支持异步保存任务，减少主线程阻塞。

### 2.3 表现与交互

- 实现基础 HUD、暂停面板、波次提示、资源面板、调试面板。
- 支持角色肖像条与英雄检查器 UI，便于查看单位推导属性和技能状态。
- 接入音频系统，统一处理音效与背景音乐播放。
- 支持资源映射表驱动的音频、纹理、字体加载。

### 2.4 稳定性与优化

- 对资源映射、关卡解析、存档写入引入异步任务，减少初始化和保存阻塞。
- 对高频日志进行降噪，保留关键错误和里程碑日志，提升调试效率。
- 修复过多处运行时崩溃点，包括技能释放链路、音效事件链路、视觉特效生命周期。
- 增加关键性能日志，如主循环、事件派发、对象数量统计等，便于后续量化验证。

### 2.5 可扩展性

- 保留渲染后端切换入口，支持 `SDL` / `OpenGL` 启动时选择，方便后续做图形后端迁移。
- 网络联机方向已经完成方案设计，能够继续扩展为服务器权威模型。
- 项目结构清晰，适合继续做关卡扩展、单位扩展、性能优化和工具链补充。

## 3. 简历可写成什么样

### 3.1 项目描述模板

**MonsterWar 2D 塔防 demo**  
`C++23` / `SDL3` / `EnTT` / `nlohmann::json` / `spdlog`

- 基于 `ECS` 与事件驱动完成单机塔防闭环，实现建造、升级、出售、波次刷怪、技能释放、胜负结算和保存读档。
- 使用数据驱动蓝图与工厂模式统一创建敌人、玩家、技能、投射物和特效，减少硬编码依赖。
- 引入异步资源映射、关卡解析与存档写入，优化启动与保存流程，降低主线程阻塞。
- 完善 HUD、暂停、波次提示、英雄检查器、调试面板等交互，并提供关键路径日志与性能统计点。

### 3.2 面试讲解模板

- 这个项目的重点不是做一个大而全的游戏，而是验证一套可扩展的 ECS 塔防架构。
- 我优先把玩法闭环做通，再对关键路径做性能和稳定性优化。
- 为了让项目更像工程项目，我把资源、蓝图、关卡、UI 统一做成数据驱动，而不是硬编码。
- 后期我补了异步加载、日志降噪和崩溃修复，确保项目既能玩，也能稳定演示。

## 4. 可供测试的内容

> 这一部分适合后续补充测试结果，用来支撑简历里的“结果驱动”表述。

### 4.1 需要测试的性能项

#### 主循环性能

- 平均 FPS
- 单帧耗时
- 波次高峰期帧耗时
- UI 打开/关闭对帧率的影响

#### 场景与加载性能

- 启动到标题页耗时
- 进入关卡耗时
- 资源映射解析耗时
- `LevelLoader` 关卡应用耗时
- 保存读档耗时

#### 事件链路性能

- `dispatcher.enqueue()` / `dispatcher.update()` 的耗时
- 单位死亡、攻击、治疗、技能释放的事件密度
- 高单位数情况下的事件积压情况

#### 渲染与特效性能

- 大量单位同时渲染时的帧时间
- 投射物、死亡特效、技能特效创建销毁成本
- 字体渲染与 HUD 刷新成本

#### 稳定性测试

- 连续运行 X 分钟无崩溃
- 关卡切换 X 次无异常
- 保存 / 回标题 / 再进入关卡流程无泄漏
- 切换渲染后端后功能保持一致

### 4.2 建议后续补充的测试格式

建议每条测试都记录：

- **测试目标**
- **测试方法**
- **测试环境**
- **测试结果**
- **结论**

例如：

- 测试目标：验证第 1 波满屏单位时的帧耗时
- 测试方法：固定地图、固定波次、重复运行 10 次
- 测试结果：平均 FPS / 1% low / 峰值耗时
- 结论：是否满足 60 FPS 目标
